Limitasi dan Skalabilitas

Asisten Pintar App

Pendahuluan

Dokumen ini menguraikan batasan teknis (limitasi) dan potensi skalabilitas dari aplikasi "Asisten Pintar". Performa dan kapasitas aplikasi ini sangat bergantung pada rate limits dari API Groq yang digunakan sebagai mesin utama untuk transkripsi, terjemahan, analisis gambar, dan pembuatan ringkasan cerdas.

Pemahaman terhadap limitasi ini sangat penting untuk merencanakan penggunaan aplikasi dalam skala yang lebih besar dan untuk mengidentifikasi strategi mitigasi yang diperlukan.

Limitasi Utama (Berdasarkan Rate Limits Groq - Akun Gratis)

Batasan penggunaan dihitung per menit. Berikut adalah rinciannya berdasarkan fungsi utama di dalam aplikasi:

1. Fungsi Peringkasan Teks (Model: Llama3-70B-8192)

Model ini digunakan saat pengguna menekan tombol "Buat Ringkasan".

  • Requests Per Minute (RPM): 30 permintaan/menit

    Artinya: Aplikasi dapat memproses hingga 30 permintaan ringkasan dalam satu menit.

  • Tokens Per Minute (TPM): 18,000 token/menit

    Artinya: Total panjang teks (gabungan dari transkrip yang dikirim dan ringkasan yang dihasilkan) tidak boleh melebihi 18,000 token (sekitar 13,500 kata) dalam satu menit.

    Implikasi Praktis: Batasan ini adalah salah satu bottleneck (penghambat) utama aplikasi. Meringkas transkrip yang sangat panjang akan menghabiskan sebagian besar kuota ini.

2. Fungsi Analisis Gambar (Model: meta-llama/llama-4-maverick-17b-128e-instruct)

Model ini digunakan saat aplikasi menganalisis konten dari screenshot atau foto yang diunggah.

  • Requests Per Minute (RPM): 30 permintaan/menit

    Artinya: Aplikasi dapat menganalisis hingga 30 gambar berbeda dalam satu menit.

  • Tokens Per Minute (TPM): 6,000 token/menit

    Artinya: Total token yang digunakan untuk memproses gambar dan menghasilkan deskripsi teks tidak boleh melebihi 6,000 token (sekitar 4,500 kata) dalam satu menit.

    Implikasi Praktis: Dengan kuota token yang lebih rendah, fungsi analisis gambar juga menjadi bottleneck signifikan, terutama jika banyak gambar dianalisis secara bersamaan.

3. Fungsi Terjemahan Teks (Model: Llama3-8B-8192)

Fungsi ini digunakan secara internal setelah transkripsi audio untuk menerjemahkan teks ke Bahasa Indonesia.

  • Requests Per Minute (RPM): 30 permintaan/menit
  • Tokens Per Minute (TPM): 30,000 token/menit

  • Keterangan: Kuota untuk terjemahan sangat longgar dan kemungkinan besar tidak akan menjadi masalah.

4. Fungsi Transkripsi Audio (Model: Whisper-Large-V3)

Fungsi ini digunakan setelah pengguna selesai merekam suara.

  • Requests Per Minute (RPM): 100 permintaan/menit

  • Keterangan: Kuota untuk transkripsi audio sangat longgar dan tidak akan menjadi masalah.

Implikasi untuk Pengguna

  • Untuk Pengguna Tunggal: Seorang pengguna individu hampir tidak mungkin mencapai batasan ini. Aplikasi akan terasa sangat cepat dan responsif.
  • Untuk Pengguna Bersamaan (Skenario Demo/Produksi): Jika aplikasi digunakan oleh banyak orang secara bersamaan, rate limit TPM (Tokens Per Minute) pada fungsi peringkasan teks dan analisis gambar adalah yang paling mungkin tercapai lebih dulu. Hal ini dapat menyebabkan beberapa pengguna mengalami sedikit keterlambatan atau pesan error sementara.

Strategi Mitigasi dan Rencana Skalabilitas

Untuk mengatasi limitasi ini dan memastikan pengalaman pengguna yang baik, berikut adalah rencana pengembangan yang dapat diimplementasikan:

  1. Pesan Status di Frontend (Jangka Pendek): Menambahkan pesan antrian yang informatif di antarmuka pengguna, seperti "Server sedang memproses permintaan lain, proses Anda akan segera dimulai...".
  2. Implementasi Sistem Antrian di Backend (Jangka Menengah): Membangun sistem antrian (misalnya menggunakan Redis atau RabbitMQ) di backend untuk mengelola permintaan ke Groq agar tidak melebihi rate limit.
  3. Upgrade Paket API (Jangka Panjang): Untuk rilis komersial, melakukan upgrade ke paket berbayar dari Groq untuk mendapatkan rate limits yang jauh lebih tinggi.

Introduction

This document outlines the technical limitations and scalability potential of the "Asisten Pintar" application. The performance and capacity of this app are highly dependent on the rate limits of the Groq API, which serves as the core engine for audio transcription, translation, image analysis, and intelligent summarization.

Understanding these limitations is crucial for planning the application's use on a larger scale and for identifying necessary mitigation strategies.

Key Limitations (Based on Groq's Free Tier Rate Limits)

Usage limits are calculated per minute. Here are the details based on the main functions within the application:

1. Text Summarization Function (Model: Llama3-70B-8192)

This model is used when the user presses the "Create Summary" button.

  • Requests Per Minute (RPM): 30 requests/minute

    Meaning: The application can process up to 30 summary requests within one minute.

  • Tokens Per Minute (TPM): 18,000 tokens/minute

    Meaning: The total length of text (the combination of the submitted transcript and the generated summary) cannot exceed 18,000 tokens (approximately 13,500 words) per minute.

    Practical Implication: This limit is one of the main bottlenecks of the application. Summarizing a very long transcript will consume a significant portion of this quota.

2. Image Analysis Function (Model: meta-llama/llama-4-maverick-17b-128e-instruct)

This model is used when the application analyzes the content of uploaded screenshots or photos.

  • Requests Per Minute (RPM): 30 requests/minute

    Meaning: The application can analyze up to 30 different images within one minute.

  • Tokens Per Minute (TPM): 6,000 tokens/minute

    Meaning: The total tokens used to process an image and generate a text description cannot exceed 6,000 tokens (approximately 4,500 words) per minute.

    Practical Implication: With a lower token quota, the image analysis function is also a significant bottleneck, especially if many images are analyzed concurrently.

3. Text Translation Function (Model: Llama3-8B-8192)

This function is used internally after audio transcription to translate text into Indonesian.

  • Requests Per Minute (RPM): 30 requests/minute
  • Tokens Per Minute (TPM): 30,000 tokens/minute

  • Note: The quota for translation is very generous and is unlikely to be an issue.

4. Audio Transcription Function (Model: Whisper-Large-V3)

This function is used after a user finishes recording audio.

  • Requests Per Minute (RPM): 100 requests/minute

  • Note: The quota for audio transcription is very generous and will not be an issue.

Implications for Users

  • For a Single User: An individual user is highly unlikely to hit these limits. The application will feel very fast and responsive.
  • For Concurrent Users (Demo/Production Scenario): If the application is used by many people simultaneously, the TPM (Tokens Per Minute) rate limit for the text summarization and image analysis functions is the most likely to be reached first. This may cause some users to experience slight delays or temporary error messages.

Mitigation Strategies and Scalability Plan

To address these limitations and ensure a good user experience, the following development plan can be implemented:

  1. Frontend Status Messages (Short-Term): Add informative queuing messages to the user interface, such as "The server is processing other requests, your process will begin shortly...".
  2. Backend Queuing System (Mid-Term): Implement a queuing system (e.g., using Redis or RabbitMQ) on the backend to manage requests to the Groq API, ensuring rate limits are not exceeded.
  3. API Plan Upgrade (Long-Term): For a commercial release, upgrade to a paid Groq API plan to obtain much higher rate limits.
